FINDING (union testing): the rejected line-regex Track treats a physical line
containing only *Unicode* whitespace (e.g. U+00A0, U+2003) -- optionally
followed by '#...' -- as an ignorable blank/comment line (its _IGNORABLE_LINE
regex uses [^\S\r\n]*, which matches all Unicode whitespace). The winning
state-machine architecture treats ONLY ASCII space (0x20) and tab (0x09) as
structural whitespace, so a Unicode-whitespace-only line becomes a data row and
a '#' after a leading Unicode space is literal content.

ASSESSMENT: this is NOT a functional bug in the winner. It is an under-specified
edge (the prompt says "blank/whitespace-only physical lines ignored" without
defining the whitespace set). The winner's choice is deliberate and, crucially,
consistent with the shared objective suite's own `unicode_not_padding` case
('\u00a0x\u00a0' is content, not padding) and with the winner's committed
`test_non_trim_whitespace_chars_preserved_in_unquoted_field`. Both finalists
pass all 64 shared bakeoff tests and every blocking sample, so no objective hard
gate is affected. Changing the winner toward the regex interpretation would be a
behavior change of uncertain correctness that touches the core scanner; it is
therefore NOT done. Instead, loop-02 closes the *coverage/documentation* gap by
committing explicit regression tests that pin this deliberate behavior, plus a
behavior-safe error-API clarity improvement (ParseError.reason). No functional
failure was manufactured.
